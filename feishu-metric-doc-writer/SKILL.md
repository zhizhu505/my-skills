# 飞书指标体系说明文档编写

## 适用场景

当用户要求为某个已从飞书 Excel 迁移到多维表格的指标体系编写说明文档时，使用本 Skill。

这类多维表格通常用于承载原 Excel 中的指标资产，下面会包含若干子目录或子表，例如：
- `指标体系`
- `指标口径`
- `指标定义&口径`
- `SQL`
- `数仓表`
- `数据基线`
- `说明文档`

本 Skill 的任务是阅读这些多维表格子目录中的内容，整理出一份 AI 可读、业务同学也能理解的指标体系说明文档。

默认目标是：优先检查是否可以编辑飞书多维表格中的说明文档；如果有权限，直接在对应说明文档中补充或修改内容；如果没有权限，向用户说明情况，并直接输出一份可粘贴到飞书文档中的完整说明文档。

不要在多维表格中新建“说明文档”视图，除非用户明确要求。说明文档通常是与“指标体系”“指标口径”“SQL”等表并列的飞书文档。

## 工作流程

1. 读取用户提供的多维表格链接、文档链接、截图或文字材料。
2. 如果有飞书 Base 链接，优先使用 `lark-cli base` 读取表、视图、字段和记录。
3. 判断多维表格中是否已有“说明文档”或类似文档。
4. 如果用户要求写入文档，先确认是否有编辑权限：
   - 有权限：直接修改或追加到已有说明文档。
   - 无权限：说明无法编辑的原因，并输出完整文档内容供用户粘贴。
5. 重点分析以下内容：
   - `指标体系`
   - `指标口径`
   - `指标定义&口径`
   - `SQL`
   - `数仓表`
   - `数据基线`
   - `文档须知`
   - `备注`
   - 其他与指标定义、维度、底层表、计算逻辑相关的表或文档
6. 忽略用户明确说明不用看的测试表、测试文档、临时视图。
7. 判断该多维表格是一套指标体系，还是包含多个分指标体系。
8. 输出或写入结构化说明文档。

## 默认文档结构

```markdown
# xxx说明文档

## 1. 原飞书文档位置

## 2. 指标体系文档说明

### 2.1 文档背景

### 2.2 指标体系结构概览

### 2.3 分析维度定义

| 维度类别 | 维度名称 | 维度定义 / 枚举值 |
| --- | --- | --- |

### 2.4 指标说明

| 指标分类 | 指标名称 | 指标定义 | 统计维度 |
| --- | --- | --- | --- |

## 3. 指标口径文档说明

### 3.1 基础表信息

### 3.2 统计范围与表粒度

### 3.3 核心计算口径与约束条件

### 3.4 分场景口径说明

### 3.5 底层表结构与字段映射

### 3.6 关联文档清单
```

可根据材料删减小节，但不要加入“使用建议”，除非用户明确要求。

## 原飞书文档位置

“原飞书文档位置”指这个多维表格来源的原 Excel 表或原始飞书文档位置，通常需要用户自己查找并补充。

如果当前材料中没有提供原 Excel/原文档位置，不要编造链接。可以保留占位：

```markdown
## 1. 原飞书文档位置

待补充：该多维表格来源的原 Excel 表或原始飞书文档位置。
```

## 指标体系文档说明

说明指标体系解决什么问题、面向什么分析场景、包含哪些指标模块。

如果一个多维表格包含多个分指标，例如“三率口径、视频监控口径、习题监控口径”，要在结构概览中体现分模块关系。指标说明可以按模块合并呈现，也可以在“指标分类”列中标明来源模块。

## 分析维度定义

维度表头固定使用：

```markdown
| 维度类别 | 维度名称 | 维度定义 / 枚举值 |
```

维度一般不会有单独文档，通常从以下位置归纳：
- 字段类型为“维度”的行
- 分类字段为“维度”的行
- 指标表中的统计维度字段
- SQL 中的 group by 字段
- 口径备注中反复出现的用户、时间、平台、班级、内容、账号、场景等限制

维度定义必须结合实际表设计动态判断，不要机械套模板。

## 指标说明

指标说明部分默认使用一张表，不要按指标分类拆很多小标题。

推荐表头：

```markdown
| 指标分类 | 指标名称 | 指标定义 | 统计维度 |
```

不要列出原子指标，除非用户明确要求。

指标定义优先来自多维表格中的“定义”“描述”“计算口径”等字段。如果没有明确文本，再根据指标名称和 SQL 谨慎概括。

## 指标口径文档说明

第三部分不要全部写成表格，也不要写成一大段。采用“分标题 + 分点 + 少量必要表格”的方式。

### 3.1 基础表信息

这里写指标计算涉及的真实底层表，而不是多维表格里的表名。

推荐写法：

```markdown
### 3.1 基础表信息

本指标体系主要依赖以下底层表进行统计：

- 核心事实表：
  - `xxx.xxx`：用于统计 xxx。
  - `xxx.xxx`：用于记录 xxx。

- 核心维度表：
  - `xxx.xxx`：用于补充用户、班级、内容、账号等维度。
```

如果材料中没有明确底层表，不要编造。可以写“当前材料未直接给出底层表名，口径主要依据现有指标定义和 SQL 说明整理”。

### 3.2 统计范围与表粒度

该部分用于说明指标统计对象、历史数据范围、更新频次、表粒度和统计周期。

优先从以下位置提取：
- `文档须知`
- `备注`
- `指标口径`
- SQL 注释
- 表字段说明
- 统计维度字段

推荐写法：

```markdown
### 3.2 统计范围与表粒度

- 用户筛选：
  - C 端、B 端访问活跃用户。

- 历史数据：
  - 自 2024-03-13 开始记录。

- 更新频次：
  - 日、周、月一更。

- 表粒度：
  - 一个活跃用户。
  - 一个用户属性。
  - 一种场景码。
  - 一条记录。

- 统计周期：
  - 支持日、周、月统计。
```

如果材料没有明确说明，可以根据口径表和 SQL 谨慎概括；如果无法判断，不要编造，直接省略对应条目或整个小节。

### 3.3 核心计算口径与约束条件

写反复出现、影响大多数指标的规则，例如：
- 时间周期
- 用户范围
- 平台限制
- 测试用户剔除
- 内容状态过滤
- 账号范围
- 24 小时窗口
- 是否去重
- 是否取最新状态
- 是否限定有效记录

示例：

```markdown
### 3.3 核心计算口径与约束条件

- 用户范围：
  - 仅统计学生用户，即 `role = 'student'`。

- 平台与端口限制：
  - 移动端通常限定 `os in ('android', 'ios')`。

- 测试用户剔除：
  - 日表通常直接剔除测试用户。
  - 周表、月表需按统计周期末最后一天状态判断。
```

### 分场景口径说明

如果指标体系包含多个业务场景，按场景写：
- 三率口径
- 视频监控口径
- 习题监控口径
- 留存口径
- 漏斗口径
- 官号数据口径
- 达人口径
- 埋点事件口径

不要写“指标与口径关系”这种机械映射，除非用户要求。

### 底层表结构与字段映射

只写关键字段，不需要复刻全表结构。

推荐写法：

```markdown
### 3.x 底层表结构与字段映射

- 活跃用户统计：
  - `u_user`：用户 ID。
  - `day`：统计日期。
  - `is_active_user`：是否活跃用户。

- 社区内容统计：
  - `id`：内容 ID。
  - `author_type`：作者类型。
  - `create_time`：内容发布时间。
```

## 口径提炼原则

1. 只保留对理解指标计算有帮助的内容。
2. 原表格中特别备注、重要约束、大部分指标共用的口径要优先写。
3. SQL 里明显的过滤条件、时间窗口、事件 key、状态字段要提炼出来。
4. 不要把所有 SQL 转成表格。
5. 不要逐条解释每个原子字段。
6. 不要编造底层表、字段、枚举、历史时间和更新频次。
7. 对不确定内容使用“根据当前材料可判断”或“当前材料未明确说明”。

## 常见特殊情况

### 多个分指标体系

如果一个多维表格包含多个分指标，应在文档中保留分模块结构。

例如：

```markdown
本指标体系由三个分指标组成：
- 课程包三率口径
- 课程包视频监控口径
- 课程包习题监控口径
```

指标表中可通过“指标分类”列标明所属模块。

### 数据基线表

数据基线表只需要概括说明，不需要逐列展开。

重点说明：
- 数据基线的时间范围
- 表粒度
- 每行代表什么
- 每列代表什么
- 与指标定义的关系

但在 3.1 基础表信息中，不要把“数据基线表”误写成底层计算表。

### 维度没有单独文档

如果维度散落在指标表中，从“分类=维度”的行、统计维度字段和 SQL group by 中归纳。

### 用户只给截图

根据截图材料输出，不要求额外读链接。不要说“信息不足”后停止；可以在文档中标注“根据当前截图材料整理”。

## 输出要求

- 默认优先尝试写入已有说明文档。
- 如果无编辑权限，说明原因，并直接输出完整说明文档。
- 使用中文。
- 结构清晰，适合直接粘贴到飞书文档。
- 不要解释冗长工作过程。
- 不要主动创建多维表格视图。
- 不要输出过长的 SQL 原文，只提炼关键表、字段和条件。
